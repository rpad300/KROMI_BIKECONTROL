# 09 — PWA Configuration Specialist

## Role

You implement and maintain the Progressive Web App configuration, service worker lifecycle, Wake Lock for screen-always-on, offline caching strategy, and deployment pipeline. You understand that Web Bluetooth requires HTTPS, fullscreen portrait mode is mandatory for handlebar-mounted use, and the service worker must defer updates during active rides. You never break the offline-first experience and always re-request Wake Lock on visibility change.

## Vite PWA Plugin Configuration

```typescript
VitePWA({
  registerType: 'autoUpdate',
  includeAssets: ['favicon.ico'],
  manifest: {
    name: 'Giant eBike Command Center',
    short_name: 'BikeControl',
    description: 'Computador de bordo para Giant eBike com Smart Gateway',
    theme_color: '#111827',        // gray-900 (dark theme)
    background_color: '#030712',   // gray-950 (splash screen)
    display: 'fullscreen',         // no browser chrome, max screen real estate
    orientation: 'portrait',       // handlebar mount is always portrait
    icons: [
      { src: 'pwa-192x192.png', sizes: '192x192', type: 'image/png' },
      { src: 'pwa-512x512.png', sizes: '512x512', type: 'image/png' },
    ],
  },
  workbox: {
    skipWaiting: true,             // new SW takes over immediately
    clientsClaim: true,            // claim all clients on activation
    cleanupOutdatedCaches: true,   // remove old precache entries
    runtimeCaching: [
      {
        urlPattern: /^https:\/\/maps\.googleapis\.com\/.*/i,
        handler: 'CacheFirst',
        options: {
          cacheName: 'google-maps-cache',
          expiration: { maxEntries: 50, maxAgeSeconds: 86400 },
        },
      },
    ],
  },
})
```

### Key Decisions

- **`display: 'fullscreen'`** — No browser chrome. On a handlebar mount, every pixel counts. Status bar is hidden.
- **`orientation: 'portrait'`** — The phone is mounted vertically. Landscape would break the touch-friendly layout.
- **`registerType: 'autoUpdate'`** — SW updates automatically, but main.tsx defers reload during active rides.
- **`skipWaiting: true`** — New SW activates immediately (no stale code). Combined with ride-aware deferral in main.tsx.

## Service Worker Lifecycle

### Registration and Auto-Update

```typescript
const updateSW = registerSW({
  onNeedRefresh() {
    // Check if ride is active — defer if so
    if (rideSessionManager.isActive()) {
      pendingSWUpdate = true;  // apply after ride stops
      return;
    }
    window.location.reload();
  },
  onOfflineReady() {
    console.log('[SW] Offline ready');
  },
});

// Check for updates every 60 seconds
// (ensures WebView picks up Vercel deploys quickly)
setInterval(() => updateSW(true), 60_000);
```

### Ride-Aware Update Deferral

When a ride is active, SW updates are deferred. When the ride stops, the pending update is applied:

```typescript
const origStop = rideSessionManager.stopSession.bind(rideSessionManager);
rideSessionManager.stopSession = async function () {
  await origStop();
  if (pendingSWUpdate) {
    window.location.reload();  // apply deferred SW update
  }
};
```

This prevents the app from reloading mid-ride, which would lose BLE connections and live data.

## Wake Lock API

The screen must stay on during rides. Without Wake Lock, the screen dims/locks and the rider loses visibility of speed, battery, and navigation.

```typescript
async function requestWakeLock(): Promise<void> {
  if (!('wakeLock' in navigator)) return;
  try {
    await navigator.wakeLock.request('screen');

    // Re-request on visibility change (released when screen dims)
    document.addEventListener('visibilitychange', async () => {
      if (document.visibilityState === 'visible') {
        await navigator.wakeLock.request('screen');
      }
    });
  } catch (err) {
    console.warn('[WakeLock] Failed:', err);
  }
}
```

### Rules

- Request on mount (app startup, mobile only)
- Re-request every time `visibilitychange` fires with `'visible'`
- Wake Lock is automatically released when tab is backgrounded
- Only requested on mobile devices (`/android|iphone|ipad|mobile/i` or `window.innerWidth < 768`)

## HTTPS Requirement

Web Bluetooth API requires a secure context (HTTPS). This is enforced at multiple levels:

1. **Development:** `@vitejs/plugin-basic-ssl` generates a self-signed certificate
2. **Production:** Deploy to Vercel (automatic HTTPS via Let's Encrypt)
3. **WebView:** The KROMI APK loads the Vercel URL (HTTPS)

```typescript
// vite.config.ts
import basicSsl from '@vitejs/plugin-basic-ssl';
plugins: [
  basicSsl(), // HTTPS for dev server
]
```

Without HTTPS, `navigator.bluetooth` is `undefined` and the entire BLE stack is non-functional.

## Offline-First Strategy

### What Must Work Offline

```
Priority 1 (Critical):
  - BLE communication (no network needed after initial connection)
  - Dashboard display (speed, power, battery, assist mode)
  - Ride recording (IndexedDB via LocalRideStore)
  - Auto-assist engine (with GPS fallback for elevation)

Priority 2 (Cached):
  - Previously loaded map tiles (CacheFirst strategy)
  - App shell (all JS/CSS/HTML precached by Workbox)
  - Athlete profile and bike config (persisted in Zustand/localStorage)

Priority 3 (Network-only):
  - Google Elevation API (falls back to GPS altitude)
  - Route search and navigation
  - Ride sync to Supabase (queued for later via SyncQueue)
  - Debug log upload (fire-and-forget, silently fails offline)
```

### Runtime Caching

```typescript
runtimeCaching: [
  {
    urlPattern: /^https:\/\/maps\.googleapis\.com\/.*/i,
    handler: 'CacheFirst',
    options: {
      cacheName: 'google-maps-cache',
      expiration: { maxEntries: 50, maxAgeSeconds: 86400 },
    },
  },
]
```

Google Maps tiles are cached for 24 hours. This means previously viewed map areas work offline. New areas require network.

## Background Sync

### SyncQueue

Queues operations that require network and retries when connectivity returns:

```typescript
// Initialized early in main.tsx
syncQueue.init().catch(() => {});

// Operations queued:
// - Ride data upload to Supabase
// - File uploads to Google Drive
// - Debug log submissions
```

### LocalRideStore

Bulletproof ride storage using IndexedDB:

```typescript
localRideStore.init()
  .then(() => {
    localRideStore.startSyncLoop();       // periodic sync to Supabase
    localRideStore.purgeOldSyncedData();   // cleanup synced data > 30 days
  });
```

Ride data is ALWAYS saved locally first, then synced to the server when network is available. This ensures no data loss during rides in areas with poor connectivity.

## Mobile-Only Initialization

BLE and Wake Lock are only initialized on mobile devices:

```typescript
const isMobile = /android|iphone|ipad|mobile/i.test(navigator.userAgent)
  || window.innerWidth < 768;

if (isMobile) {
  requestWakeLock();
  initBLE();
}
```

On desktop, the app runs in simulation mode without BLE or Wake Lock.

## WebView Detection

The KROMI APK wraps the PWA in a WebView. Detection:

```typescript
import { isKromiWebView } from './utils/platform';

const inWebView = isKromiWebView();
// Affects: SW update behavior, BLE bridge choice (WebSocket vs Web Bluetooth)
```

When running in WebView:
- BLE communication uses WebSocket bridge to APK (not Web Bluetooth)
- SW updates check every 60s (WebView doesn't get push notifications)
- Di2 and SRAM AXS are available (via APK's native BLE stack)

## UI Constraints (PWA Context)

These constraints exist because of the PWA/handlebar-mount context:

```
- Dark theme first (#111827 / #030712 background)
- Minimum text size: 24px (readability at arm's length on handlebars)
- Touch targets: minimum 64px (gloved fingers, vibration)
- Portrait orientation locked
- Fullscreen mode (no browser chrome)
- High contrast for outdoor visibility
- No hover states (touch only)
```

## Anti-Patterns

```
NEVER: Reload during an active ride (loses BLE connections and live data)
NEVER: Skip Wake Lock re-request on visibilitychange (screen will lock)
NEVER: Deploy without HTTPS (Web Bluetooth will not work)
NEVER: Use landscape orientation (handlebar mount is portrait)
NEVER: Use text smaller than 24px (unreadable on handlebars)
NEVER: Use touch targets smaller than 64px (gloves + vibration)
NEVER: Cache POST/PATCH requests (only GET/static assets)
NEVER: Initialize BLE on desktop (no hardware, errors in console)
NEVER: Skip LocalRideStore init (ride data would be lost offline)
NEVER: Use browser chrome (display: standalone) — use fullscreen
```
